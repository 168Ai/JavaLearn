这份文档是一份关于**ELK（Elasticsearch, Logstash, Kibana）技术栈实战部署与应用**的课堂实录讲义。内容涵盖了从环境准备、Docker编排部署、数据同步（Canal）到可视化展示的完整流程。

以下是该课程内容的详细总结：

### 1. 核心概念与架构 (ELK Overview)

课程首先介绍了ELK三个核心组件及其功能：

* **Elasticsearch (ES)**：分布式全文检索引擎。核心原理是**分词**（如IK分词器），适合海量数据的快速搜索，性能高因为数据常驻内存。
* **Logstash**：数据处理中间件。负责数据的**清洗、过滤、格式转换**（如从MySQL同步数据时去除无用字段）。
* **Kibana**：数据展示层。提供图表化（折线图、饼图等）和BI报表功能。
* **应用场景**：
  * **日志收集**：解决多服务器日志查询困难的问题。
  * **异构数据同步**：将MySQL数据同步到ES，实现商品搜索的自动补全、模糊搜索等功能。

### 2. 环境准备与部署 (Environment & Deployment)

讲师详细演示了基于**Docker**和**Docker Compose**的集群部署方案：

* **基础设施**：推荐使用Linux虚拟机，内存至少8G（生产环境建议更高），需关闭Swap分区以保证性能。
* **工具链**：
  * **Docker Compose**：作为单机版任务编排工具，用于统一管理多个容器（ES节点、Kibana、MySQL、RabbitMQ等）的启停。
  * **网络配置**：使用Docker内部网络（Bridge），容器间通过服务名（域名）互相访问，仅对外暴露必要端口。
* **ES集群配置**：
  * 包含3个节点（node-1, node-2, node-3），配置了集群名称、节点名称、数据存储路径及内存锁定。
  * 需安装**IK分词器**插件以支持中文分词。

### 3. 数据同步流程 (Data Synchronization)

课程重点讲解了如何将MySQL数据实时同步至ES，构建搜索库：

* **同步原理**：利用**Canal**组件伪装成MySQL的Slave，通过解析MySQL的**Binlog**日志（需开启Row模式）获取数据变更。
* **中间件角色**：
  * **RabbitMQ**：作为消息缓冲队列，解耦Canal和Logstash。
  * **Logstash**：消费队列中的消息，进行数据清洗（如移除`_class`、`server_id`等无用字段，处理数组结构，转换时间格式）。
* **数据流向**：MySQL → Canal → RabbitMQ → Logstash → Elasticsearch。

### 4. 数据清洗与索引管理 (Data Processing)

* **Logstash配置**：分为`input`（来源）、`filter`（过滤处理）、`output`（输出）三部分。
  * **清洗逻辑**：使用Ruby语法处理字段，将嵌套在`data`数组中的实际业务数据提取到根层级，并删除原始数组。
  * **坐标处理**：将经纬度拼接为Geo-point类型，支持地图可视化。
* **索引模板 (Index Template)**：
  * 建议按日期创建索引（如`logs-cm-%{+YYYY.MM.dd}`），利用**别名 (Alias)** 进行统一查询，解决单索引数据量过大和不可扩容的问题。

### 5. 可视化分析 (Visualization)

* **Kibana操作**：
  * 创建**索引模式 (Index Pattern)** 以关联ES中的数据。
  * **Discover**：用于查看原始数据流和时间轴分布。
  * **Dashboard**：构建可视化仪表盘，演示了以下图表：
    * **柱状图**：统计各省份小区总户数（聚合求和）。
    * **折线图**：分析历年小区建筑面积趋势（反映房地产高峰）。
    * **树状图**：展示各省各市的绿化率分布。
    * **地图**：基于地理坐标（Geo-point）展示小区在全国的分布密度。

### 6. 实战案例与性能调优

* **案例数据**：导入了全国30多万条真实的小区数据（包含经纬度、户数、绿化率等）。
* **硬件建议**：强调了内存的重要性，指出ES是内存密集型服务，建议开发测试环境至少4G，推荐8G或以上。
* **部署策略**：建议将“ES集群”与“同步组件（MySQL/Canal等）”分开部署，以便在数据导入时停用同步组件释放内存，防止内存溢出。

# ELK 平台实战课程 完整总结

## 一、ELK 是什么
ELK 是一套开源日志与数据处理解决方案，由三个核心组件组成：
1. **Elasticsearch（ES）**
   - 分布式**全文检索引擎**
   - 支持分词、关键词搜索、聚合统计
   - 性能远优于 MySQL 模糊查询（`like %xx%`）
2. **Logstash**
   - 数据**采集、清洗、转换、过滤**中间件
   - 负责格式化日志、剔除无用字段、拆分批量数据
3. **Kibana**
   - 可视化平台，以图表、报表、地图方式展示 ES 数据
   - 支持柱状图、折线图、热力图、地图、仪表盘等

> 现在官方叫 **ELK Stack**，常搭配 **Filebeat、Kafka、RabbitMQ、Canal** 一起使用。

---

## 二、ELK 典型应用场景
### 1. 分布式系统日志统一收集
- 传统：一台台服务器登录查日志，效率极低
- ELK 方案：
  1. 每台机器部署 **Filebeat** 采集日志
  2. 发送到 Kafka/RabbitMQ 缓冲
  3. Logstash 清洗过滤
  4. 存入 ES
  5. Kibana 统一搜索、查看、告警

### 2. 异构数据同步（MySQL → ES）
- MySQL 不适合全文搜索、模糊搜索、高并发检索
- 通过 **Canal** 监听 MySQL binlog，同步数据到 ES
- 实现商品搜索、用户检索、地址检索等业务

### 3. 数据可视化与 BI 报表
- 流量统计、接口错误率、用户行为分析
- 按时间、地区、类型生成图表
- 实时监控业务指标

## 三、本次实战架构（MySQL → ES 异构同步）
整体流程：
1. **MySQL** 产生业务数据，开启 binlog
2. **Canal** 伪装成 MySQL 从库，抓取 binlog
3. 数据发送到 **RabbitMQ** 做缓冲削峰
4. **Logstash** 消费消息，做数据清洗
5. 清洗后写入 **Elasticsearch**
6. **Kibana** 做可视化展示

组件清单：
- MySQL 5.7
- Canal
- RabbitMQ
- Logstash
- Elasticsearch 集群（3 节点）
- Kibana
- cerebro（ES 集群监控面板）

---

## 四、环境准备与系统优化
### 1. 机器要求
- 内存至少 **8G**，否则 ES 无法稳定运行
- CPU 要求不高
- 必须安装：Docker + Docker Compose

### 2. 关键系统配置（必须做）
1. **提高文件句柄数**
   ES 大量读写小文件，默认句柄不足
2. **关闭 swap 分区**
   生产必须关，否则 ES 性能急剧下降
3. **调大线程数**
   避免启动报错“线程不足”
4. **挂载目录授权**
   Docker 映射数据目录需权限

---

## 五、ES 集群部署（Docker Compose）
### 1. 3 节点集群配置要点
- 集群名统一
- 节点名不同
- `node.master: true`：候选主节点
- `node.data: true`：数据节点
- 通信端口：9200（HTTP）、9300（内部）
- 集群通过**服务名（域名）**互相发现

### 2. 必装插件
- **IK 分词器**
  用于中文分词，版本必须与 ES 严格一致

### 3. Docker Compose 作用
- 统一编排多个容器
- 一条命令启动/停止整个集群
- 容器间用自定义 Docker 网络互通，用服务名访问

## 六、Canal + RabbitMQ 数据同步部署
### 1. MySQL 配置
- 开启 binlog
- 设置 `binlog_format = ROW`
- 创建 Canal 专用账号并授权

### 2. Canal 作用
- 监听 MySQL 数据增删改
- 解析 binlog 转为结构化数据
- 投递到 RabbitMQ
- 支持集群模式（ZK 选主，保证高可用）

### 3. RabbitMQ 配置
- 创建交换机、队列
- 与 Logstash 配置保持一致
- 持久化确保消息不丢失

---

## 七、Logstash 数据清洗核心
Logstash 配置三部分：
1. **input**：数据源（RabbitMQ）
2. **filter**：数据清洗（核心）
3. **output**：输出目标（ES / 控制台）

### 本次清洗逻辑（重点）
1. 丢弃**删除操作**的数据
2. 拆分批量 data 数组 → 拆成单条记录
3. 删除无用字段（database、table、pk 等）
4. 时间字段格式化、时区校正
5. 经纬度组合为 ES 地理坐标类型
6. 把嵌套 data 内字段**提升到顶层**
7. 最终输出到 ES 按天生成索引

---

## 八、ES 索引设计
### 1. 按天建索引
- 格式：`logs_小区_2026-04-14`
- 优点：避免单索引过大，便于冷热分离、删除历史数据

### 2. 索引模板（Index Template）
统一设置：
- 主分片 / 副本分片
- mappings（字段类型）
  - text：分词搜索
  - keyword：不分词，精确匹配
  - date：日期
  - float / integer：数值
  - geo_point：地理坐标

### 3. 索引别名
- 多个按天索引共用一个别名
- 查询时只需查别名，自动跨索引搜索

## 十、关键问题与坑点总结
1. **ES 非常吃内存**，8G 是起步，2G 基本跑不起来
2. IK 分词器版本必须与 ES 一致
3. Canal 必须开启 binlog_row_image
4. Logstash 过滤规则写错会导致数据结构混乱
5. 容器网络不通 → 检查 Docker network
6. 索引 mapping 设计不当会导致无法搜索/聚合
7. 生产必须关闭 swap，否则极不稳定
