# 三、IO 本质：数据拷贝
IO = **计算机内存与外部设备之间的数据拷贝**。

应用程序不能直接操作网卡/磁盘，必须通过**操作系统内核**。

完整流程：
1. 网卡 → 内核缓冲区（内核空间）
2. 内核缓冲区 → 应用内存（用户空间）

**两次拷贝 = IO**
- 读：网卡 → 内核 → 用户
- 写：用户 → 内核 → 网卡

---

# 四、用户态 & 内核态
- **用户态**：应用程序运行，权限低，不能直接操作硬件
- **内核态**：操作系统运行，权限高，管理网卡、内存、CPU

系统调用（read/write/socket）会触发**用户态 ↔ 内核态切换**。

---

# 五、Socket 是什么
Socket = **操作系统提供的网络编程 API 集合**，用于跨主机进程通信。

1. Linux 一切皆文件，Socket 也是文件
2. 每个 Socket 对应一个 **文件描述符 fd（int 数字）**
3. 应用通过 fd 告诉内核：操作哪个连接

---

# 六、网络编程底层流程（系统调用视角）
## 服务端
1. `socket()`：创建服务端 Socket（fd=3）
2. `bind()`：绑定 IP + 端口
3. `listen()`：开启监听
4. `accept()`：阻塞等待客户端连接
   内核每接受一个新连接，会**新建一个 Socket**（如 fd=5）代表该客户端
5. `read()/write()`：通过 fd 读写数据

## 客户端
1. `socket()`：创建客户端 Socket
2. `connect()`：发起 TCP 三次握手
3. `read()/write()`：收发数据

# 八、IO 模型核心分类（本节课重点）
## 1. 同步阻塞 IO（BIO）
### 特点
- **同步**：应用必须主动调用 read 才能拿到数据
- **阻塞**：调用 read 时缓冲区没数据 → 线程卡死等待

### 问题
- **一个连接占一个线程**
- 线程资源浪费严重
- 高并发下直接崩

### 典型表现
`accept()` 阻塞、`read()` 阻塞。

---

## 2. 同步非阻塞 IO（NIO 雏形）
通过 `fcntl` 设置 Socket 为 **非阻塞**。

### 特点
- 调用 `read()` 没数据 → **直接返回，不等待**
- 应用必须**轮询**反复调用 read

### 优点
- 线程不会长时间阻塞
- 线程利用率略微提升

### 缺点
- **大量无效系统调用**
- 仍然一个连接绑定一个线程
- 轮询消耗 CPU

---

## 3. IO 多路复用（本节课最重要）
**解决核心问题：不知道数据什么时候来。**

思路：
**先检测哪些 Socket 有数据，有数据再去 read。**

一次系统调用可以**同时监听成千上万个 Socket**。

### 典型实现
- select
- poll
- epoll（Linux 最优）
- kqueue（BSD/Mac）

---

# 九、select 模型详解
## 工作流程
1. 应用维护一个 fd 集合（要监听的 Socket）
2. 调用 `select`，把集合传给内核
3. 内核遍历所有 fd，检查是否就绪
4. select 返回，应用遍历集合找就绪 fd
5. 对就绪 fd 执行 `accept()` / `read()`

## 优点
- **一个线程监听大量连接**
- 线程资源极大节省
- 跨平台支持好

## 缺点
- 默认最多监听 1024 个 fd
- 每次调用需要**从用户空间拷贝 fd 集合到内核**
- 返回后需要**遍历全部 fd**找就绪者
- 连接越多效率越低

---

# 十、poll 模型
- 是 select 的改进版
- 无 1024 限制
- 原理仍类似：遍历 + 全量拷贝
- 效率仍不高

---

# 十一、本节课总结（最精简版）
1. **IO = 数据拷贝**，必须经过内核
2. **Socket = 网络连接的抽象，用 fd 标识**
3. **BIO 阻塞严重，一连接一线程，扛不住高并发**
4. **非阻塞 IO 靠轮询，仍浪费资源**
5. **IO 多路复用 = 先检测再读写，一个线程管理大量连接**
6. **select/poll 效率一般，epoll 才是高性能关键（下节课讲）**
